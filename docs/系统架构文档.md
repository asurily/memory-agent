# Memory Agent — 系统架构文档

> 版本: v2.0 | 最后更新: 2026-05-02

## 1. 系统定位

Memory Agent 是一个**多轮对话记忆中间件**，为 AI Agent 提供会话管理、用户画像、意图总结、记忆存储、上下文压缩和安全扫描能力。它不是 Agent 框架本身，而是 Agent 框架的**记忆子系统**，可以嵌入任意 Java / Spring Boot 项目。

**核心价值：** 让 AI Agent "记住" 用户，跨会话保持上下文一致性。

**设计哲学：**
- **能力默认启用 > 配置开关** — 所有能力默认启用，LLM 相关功能自动检测基础设施后启用，应用方只需提供 LLM apiKey
- **SPI 优先 > 硬编码实现** — 存储、LLM、向量嵌入、向量存储全部可插拔
- **不可变值对象 > 可变状态** — MemoryEntry / ConversationMessage 等均为不可变，修改返回新实例
- **零框架依赖** — core 模块仅依赖 SLF4J，纯 Java 17 SDK

## 4. 核心子系统

### 4.1 MemoryAgent（门面）

**文件：** `MemoryAgent.java`（~750 行）

唯一的用户入口，Builder 模式创建。职责：
- 聚合所有子系统实例
- 所有组件始终创建，LLM 相关功能通过 `hasLlm` 检测自动启用
- 暴露 35+ 个声明式 API
- 管理 Session 级冻结快照 (`ConcurrentHashMap<sessionId, MemorySnapshot>`)
- 提供 `getMetrics()` 暴露不可变性能指标快照
- 自动管道编排（auto-pipeline），守护线程驱动异步总结与压缩

**hasLlm 检测逻辑：**

```
hasLlm = (config.getLlmApiKey() != null && !config.getLlmApiKey().isBlank())
```

当 `hasLlm = true` 时，以下能力自动启用：
- 意图总结（IntentSummarizer）
- 自动记忆提取（MemoryExtractor）
- 冲突检测（MemoryConflictDetector）

当 `hasLlm = false` 时：
- 使用 NoOpIntentSummarizer（总结返回空结果）
- 冲突检测跳过，默认 KEEP_BOTH
- 自动记忆提取跳过

**创建流程：**

```
builder.build()
  ├─ resolveStorage()              → SPI 或 InMemoryMemoryStorage
  ├─ resolveSummarizer()           → OpenAIIntentSummarizer 或 NoOpIntentSummarizer（取决于 hasLlm）
  ├─ EmbeddingService              → SPI 或 SimpleEmbeddingService（始终创建）
  ├─ VectorStore                   → SPI 或 InMemoryVectorStore（始终创建）
  ├─ KeywordSearchService          → SPI 或 InMemoryKeywordSearchService（始终创建）
  ├─ RelevanceTracker              → 始终创建
  ├─ MemoryMetricsCollector        → 始终创建
  ├─ MemoryConflictDetector        → 始终创建（hasLlm 时启用 LLM 检测，否则跳过）
  ├─ conflictExecutor              → ASYNC 模式时创建（daemon single thread）
  ├─ CuratedMemoryManager（注入全部依赖）
  ├─ MemoryExtractor               → 始终创建（hasLlm 时启用 LLM 提取，否则跳过）
  ├─ IntentSummarizationEngine（注入 MemoryExtractor）
  ├─ ContextCompressor
  ├─ MemoryTierManager
  ├─ MemoryDecayEngine + 定时任务   → 始终创建
  └─ pipelineExecutor              → daemon 单线程 "auto-pipeline"，始终创建
```

**TurnContext 扩展：**

每次对话轮次内部使用 `TurnContext` 传递上下文，包含 `pendingSummarization` 标记，用于自动管道编排。

### 4.2 CuratedMemoryManager（记忆 CRUD 核心）

**文件：** `memory/CuratedMemoryManager.java`（~380 行）

所有记忆写入的核心路径。写入流程：

```
addEntry(userId, content, type)
  ├─ 1. SecurityScanner.scanAndThrow(content)
  ├─ 2. MemoryEntry.of(content, type)  → 不可变对象
  ├─ 3. 去重检查（精确匹配 content）
  ├─ 4. [有 LLM 时] 冲突检测（按模式分支）
  │    ├─ SYNC:  handleConflictSync() → 等待 LLM 结果，修复 REPLACE_OLD/MERGE 逻辑
  │    └─ ASYNC: handleConflictAsync() → 立即写入 + 提交后台任务修正
  ├─ 5. 容量检查（仅计算 CORE 层）
  ├─ 6. storage.addMemoryEntry()
  ├─ 7. embeddingService.embed() + vectorStore.upsert()（始终执行）
  ├─ 8. keywordSearchService.index()（始终执行）
  └─ 9. metricsCollector.recordAddMemory()（始终执行）
```

**冲突检测模式：**

| 模式 | 行为 | 适用场景 |
|------|------|---------|
| `SYNC`（默认） | 写入前等待 LLM 检测结果，决定是否写入 | 需要强一致性 |
| `ASYNC` | 先写入，后台检测冲突后修正（DISCARD 回滚/REPLACE 删除旧/MERGE 合并） | 低延迟优先 |

**修复逻辑：**
- `REPLACE_OLD` → 根据 `replacedEntryId` 或内容匹配，删除旧条目
- `MERGE` → 删除旧条目，写入 `mergedContent`

**其他方法：**
- `searchWithRelevanceTracking()` — 语义搜索 + 自动提升高频检索记忆评分
- `removeEntry()` — 同步清理向量存储、关键词索引、相关性跟踪

**冻结快照：** `captureSnapshot()` 只返回 CORE 层记忆，过滤掉 ARCHIVED 层。

### 4.3 IntentSummarizationEngine（意图总结）

**文件：** `summary/IntentSummarizationEngine.java`（~130 行）

编排 LLM 总结流程：
1. 加载会话消息 + 用户画像 + 前序摘要
2. 调用 `IntentSummarizer.summarize()` SPI
3. 持久化 `IntentSummary` 到存储
4. 触发 `MemoryExtractor.extractAndStore()`（有 LLM 时自动启用）

### 4.4 ContextCompressor（上下文压缩）

**文件：** `compression/ContextCompressor.java`

三阶段管线：工具输出裁剪 → 头尾保护 → LLM 中间摘要。保护 LLM Context Window 不溢出。

### 4.5 SecurityScanner（安全扫描）

**文件：** `security/SecurityScanner.java`

三层防护：不可见 Unicode → 提示注入正则 → 数据泄露/后门正则。所有记忆写入和内容总结前自动调用。

## 5. 高级子系统

### 5.1 自动记忆提取

**新增文件：** `summary/MemoryExtractor.java`, `summary/MemoryExtractionPrompt.java`, `model/MemoryExtraction.java`

**触发时机：** `IntentSummarizationEngine.summarize()` 完成后（有 LLM 时自动启用）

**流程：**

```
extractAndStore()
  ├─ summarizer.extractMemories(messages, existingProfile)
  │    └─ LLM 返回 JSON 数组 [{content, type, confidence, category}]
  ├─ 过滤 confidence < threshold 的条目
  ├─ 逐条安全扫描
  ├─ 去重检查
  └─ 写入 storage
```

**SPI 扩展点：** `IntentSummarizer.extractMemories()` — default 方法返回空列表，不影响已有实现。

### 5.2 语义搜索

**新增文件：** `spi/EmbeddingService.java`, `spi/VectorStore.java`, `model/VectorSearchResult.java`, `spi/defaults/SimpleEmbeddingService.java`, `spi/defaults/InMemoryVectorStore.java`

**始终启用（默认 SimpleEmbeddingService）**。写入记忆时自动向量化，搜索时自动语义匹配。

**写入时向量化：** 在 `CuratedMemoryManager.addEntry()` 中，始终执行 embeddingService.embed() + vectorStore.upsert()。

**搜索时：** `MemoryAgent.searchMemories()` → embed(query) → vectorStore.search(topK, filter)

**默认实现：**
- `SimpleEmbeddingService` — 基于 SHA-256 哈希的词袋模型，零外部依赖，适合开发测试
- `InMemoryVectorStore` — ConcurrentHashMap + 遍历余弦相似度，适合开发测试

**生产替换：** 实现 `EmbeddingService` / `VectorStore` 接口，通过 Builder 或 Spring Bean 注入。

### 5.3 记忆衰退与淘汰

**新增文件：** `memory/MemoryDecayStrategy.java`, `memory/TimeBasedDecayStrategy.java`, `memory/CompositeDecayStrategy.java`, `memory/MemoryDecayEngine.java`

**始终启用。** 衰退引擎在 MemoryAgent 构建时自动启动定时任务。

**评分策略：**

| 策略 | 公式 | 适用场景 |
|------|------|---------|
| TimeBasedDecayStrategy | `importance × e^(-λ × days)` | 简单场景 |
| CompositeDecayStrategy | `0.4×time + 0.3×freq + 0.3×recency` | 综合评估（默认） |

**衰退引擎：**
- `startScheduledDecay(intervalMinutes)` — 启动守护线程定时执行
- `runDecayCycle(userId)` — 评估所有 CORE 记忆，低于阈值的降级到 ARCHIVED
- `importanceScore >= protectionThreshold` 的记忆受保护，不被动淘汰

**联动：** 衰退不删除，而是降级到 ARCHIVED 层，可通过语义搜索找回。

### 5.4 记忆冲突检测

**新增文件：** `model/ConflictResolution.java`, `memory/MemoryConflictDetector.java`

**集成点：** `CuratedMemoryManager.addEntry()` 中去重检查之后、容量检查之前。

**有 LLM 时自动启用。** 无 LLM 时跳过冲突检测，默认 KEEP_BOTH 行为。

**流程：**

```
detect(newContent, existingEntries)
  ├─ summarizer.detectConflict(newContent, existing)
  │    └─ LLM 返回 {conflictType, resolution, mergedContent, confidence, explanation, replacedEntryId}
  └─ 返回 ConflictResolution
       ├─ CONTRADICT → REPLACE_OLD
       ├─ UPDATE → REPLACE_OLD
       ├─ DUPLICATE → DISCARD_NEW
       └─ SUPPLEMENT → KEEP_BOTH
```

**v1.2 增强：**
- `replacedEntryId` 字段 — LLM 明确指定要替换/合并的旧条目 ID，支持精准操作
- REPLACE_OLD 和 MERGE 分支的缺失逻辑已修复：实际删除旧条目、写入合并内容
- 新增 ASYNC 模式（见 5.7）

**SPI 扩展点：** `IntentSummarizer.detectConflict()` — default 方法返回 SUPPLEMENT/KEEP_BOTH。

**ConflictResolution 数据模型：**

```
ConflictResolution
├── conflictType: CONTRADICT | SUPPLEMENT | DUPLICATE | UPDATE
├── resolution: REPLACE_OLD | MERGE | KEEP_BOTH | DISCARD_NEW
├── mergedContent: String (MERGE 时填充)
├── confidence: double
├── explanation: String
└── replacedEntryId: String (REPLACE_OLD/MERGE 时填充)
```

### 5.5 分层记忆架构

**新增文件：** `model/MemoryTier.java`, `memory/MemoryTierManager.java`

**三层模型：**

| 层级 | 枚举值 | 注入方式 | 检索方式 |
|------|--------|---------|---------|
| L1 CORE | `MemoryTier.CORE` | 始终注入 Prompt | 直接获取 |
| L2 ARCHIVED | `MemoryTier.ARCHIVED` | 不注入 | 语义搜索 |
| L3 RAW | `MemoryTier.RAW` | 不注入 | 按需检索 |

**迁移 API：**
- `archiveMemory()` — CORE → ARCHIVED
- `promoteMemory()` — ARCHIVED → CORE
- `runDecayCycle()` — 自动 CORE → ARCHIVED（衰退联动）

**影响范围：**
- `captureSnapshot()` 只包含 CORE 层
- 容量检查只计算 CORE 层
- `MemoryEntry.tier` 默认为 CORE，向后兼容

### 5.6 混合检索

**新增文件：** `spi/KeywordSearchService.java`, `spi/defaults/InMemoryKeywordSearchService.java`, `memory/HybridSearchStrategy.java`

**始终启用。** 向量搜索与关键词搜索始终同时执行，结果通过 RRF 融合。

**流程：**

```
searchMemories(query)
  ├─ vectorStore.search(queryEmbedding)        → 向量结果
  ├─ keywordSearchService.search(query)         → BM25 关键词结果
  └─ HybridSearchStrategy.rrf(vector, keyword)  → RRF 融合
       └─ score = Σ(1 / (k + rank_i)), k=60
```

**默认 BM25 实现（InMemoryKeywordSearchService）：**
- 参数：k1=1.2, b=0.75
- 分词：按非字母数字分割，转小写
- 索引生命周期：`index()` 在 addEntry 中同步，`remove()` 在 removeEntry 中同步
- 线程安全：ConcurrentHashMap

**SPI 扩展点：** `KeywordSearchService` — 可替换为 Elasticsearch、Meilisearch 等。

### 5.7 异步冲突检测

**门控：** `conflictDetectionMode=SYNC`（默认）/ `ASYNC`（可配置）

**ASYNC 模式流程：**

```
addEntry(newEntry)
  ├─ 立即写入 storage + 向量化 + 关键词索引
  ├─ 返回新条目（不等待 LLM）
  └─ [后台] conflictExecutor.submit()
       ├─ detectConflict(newContent, existing)
       ├─ DISCARD_NEW → removeEntry() 回滚
       ├─ REPLACE_OLD → removeEntry(replacedEntryId) 删除旧条目
       ├─ MERGE → removeEntry(old) + addEntry(mergedContent)
       └─ KEEP_BOTH → 不做任何操作
```

**线程模型：** daemon 单线程执行器 (`conflict-detection`)，JVM 退出时自动回收。

### 5.8 记忆重要性自学习

**新增文件：** `memory/RelevanceTracker.java`

**始终启用。** 检索频率自动跟踪并提升高频记忆的 importanceScore。

**双通道机制：**

| 通道 | 触发时机 | 行为 |
|------|---------|------|
| LLM 提取评分 | `extractMemories()` | LLM 输出 `importanceScore` (0.0-1.0)，指导按评分指导 |
| 检索频率提升 | `searchMemories()` | 每 5 次检索自动+0.05，上限 1.0 |

**LLM 重要性评分指导：**

| 评分范围 | 含义 |
|---------|------|
| 0.9-1.0 | 关键身份信息（姓名、角色、核心约束） |
| 0.6-0.8 | 重要偏好、习惯、长期目标 |
| 0.3-0.5 | 一般事实和次要细节 |
| 0.0-0.2 | 临时性信息 |

**RelevanceTracker 参数：**
- `BOOST_INTERVAL = 5` — 每 5 次检索触发提升
- `BOOST_AMOUNT = 0.05` — 每次提升量
- `MAX_IMPORTANCE = 1.0` — 上限

### 5.9 性能指标监控

**新增文件：** `memory/MemoryMetricsCollector.java`

**始终启用。** 所有操作自动记录指标，零配置。

**收集指标：**

| 指标 | 类型 | 说明 |
|------|------|------|
| addMemoryCount | 计数器 | 写入次数 |
| searchCount | 计数器 | 搜索次数 |
| summarizeCount | 计数器 | 总结次数 |
| decayRunCount | 计数器 | 衰退轮次 |
| memoryExtractionCount | 计数器 | 自动提取次数 |
| errorCount | 计数器 | 错误次数 |
| addMemoryLatencyNs | 延迟累加 | 写入延迟 |
| searchLatencyNs | 延迟累加 | 搜索延迟 |
| searchHitCount / searchMissCount | 命中率 | 搜索结果非空/空的比例 |
| conflictDistribution | 冲突分布 | 各 resolution 类型计数 |

**API：** `MemoryAgent.getMetrics()` → `MemoryMetricsCollector.MetricsSnapshot`（不可变快照）

**生命周期：** `shutdown()` 时自动打印汇总指标到日志。

**技术选型：** `LongAdder` + `ConcurrentHashMap`，零外部依赖，无性能开销。

### 5.10 自动管道编排（v2.0 新增）

v2.0 引入自动管道编排机制，在对话轮次中自动触发总结和压缩，无需应用方手动调用。

**管道触发点：**

| 触发点 | 自动操作 | 执行模式 |
|--------|---------|---------|
| `processUserMessage()` | auto-summarize | ASYNC（后台线程） |
| `recordAssistantMessage()` | auto-compress | ASYNC（后台线程） |

**核心机制：**

```
TurnContext（每轮次上下文）
  └── pendingSummarization: boolean
       └─ processUserMessage() 中检测消息数/token 数超过阈值时置为 true
           └─ recordAssistantMessage() 结束时触发异步总结

pipelineExecutor（SingleThreadExecutor，守护线程 "auto-pipeline"）
  ├─ 始终在 MemoryAgent 构建时创建
  ├─ 提交异步总结任务
  └─ 提交异步压缩任务

summarizationInProgress: ConcurrentHashMap<sessionId, Boolean>
  └─ 防止同一会话重复触发总结（去重保护）

autoCompressionContextWindowTokens（默认 8000）
  └─ 自动压缩的 Context Window 阈值
  └─ 当累计 token 数超过此值时触发自动压缩
```

**管道流程：**

```
processUserMessage(sessionId, content)
  ├─ 安全扫描
  ├─ 存储消息
  ├─ 检查总结阈值（消息数 / token 数）
  └─ 设置 pendingSummarization = true（超过阈值时）

recordAssistantMessage(sessionId, content)
  ├─ 安全扫描
  ├─ 存储消息
  ├─ [ASYNC] 如果 pendingSummarization=true → pipelineExecutor.submit(summarize)
  │    └─ summarizationInProgress.put(sessionId, true) 防重复
  └─ [ASYNC] 检查 token 窗口 → pipelineExecutor.submit(compress)（超过阈值时）
```

## 8. 配置体系

### 8.1 配置项

v2.0 移除了所有功能 boolean 开关，能力全部自动启用。应用方只需提供 LLM apiKey 即可自动激活全部高级功能。

```yaml
memory-agent:
  # ====== 基础配置 (v1.0) ======
  enabled: true
  storage-type: in-memory
  memory-char-limit: 2200
  user-profile-char-limit: 1375
  summarization-message-threshold: 10
  summarization-token-threshold: 4000
  summarize-on-session-close: true
  compression-threshold-ratio: 0.5
  compression-protect-first-n: 3
  security-scan-enabled: true
  llm:
    api-key: null                      # 提供 apiKey 即自动启用全部 LLM 相关功能
    model: gpt-4o-mini
    base-url: https://api.openai.com/v1
    max-tokens: 2000
    temperature: 0.3
    timeout-seconds: 30
    proxy-host: null
    proxy-port: 0

  # ====== 高级配置（参数调优，非开关） ======
  memory-extraction-confidence-threshold: 0.7
  embedding-dimension: 1536
  vector-store-type: in-memory
  decay-half-life-days: 30
  decay-threshold: 0.1
  decay-schedule-interval-minutes: 60
  importance-protection-threshold: 0.8
  conflict-detection-mode: SYNC       # SYNC | ASYNC

  # ====== v2.0 新增配置 ======
  auto-compression-context-window-tokens: 8000   # 自动压缩的 Context Window 阈值
```

> **注意：** v1.1/v1.2 的 boolean 开关（如 `auto-memory-extraction-enabled`、`semantic-search-enabled`、`decay-enabled`、`conflict-detection-enabled`、`hybrid-search-enabled`、`auto-importance-learning-enabled`、`metrics-enabled`）已被移除。在 application.yml 中保留这些配置项不会报错，但会被静默忽略。

### 8.2 配置流

```
Spring Boot: memory-agent.* → MemoryAgentProperties → MemoryAgentConfig.builder()
纯 Java:     直接使用 MemoryAgentConfig.builder()
                                                      ↓
                                              MemoryAgent(Builder)
                                                ├─ hasLlm = (apiKey != null && !apiKey.isBlank())
                                                ├─ 有 LLM → 全部 LLM 功能自动启用（总结、提取、冲突检测）
                                                ├─ 无 LLM → NoOpIntentSummarizer，跳过冲突检测
                                                ├─ 向量化/关键词索引/衰退/指标 → 始终启用
                                                └─ pipelineExecutor → 始终创建，守护线程驱动
```

## 9. 线程模型

| 组件 | 线程安全机制 |
|------|------------|
| `InMemoryMemoryStorage` | `ConcurrentHashMap` + `compute()` 原子操作 |
| `InMemoryVectorStore` | `ConcurrentHashMap` |
| `MemoryAgent.sessionSnapshots` | `ConcurrentHashMap` |
| `MemoryDecayEngine` | `SingleThreadScheduledExecutor`（守护线程 `memory-decay-engine`） |
| `summarizeAsync()` | `CompletableFuture.supplyAsync()`（ForkJoinPool） |
| `MemoryEntry` | 不可变值对象，天然线程安全 |
| `MemoryConflictDetector` (ASYNC) | `SingleThreadExecutor`（守护线程 `conflict-detection`） |
| `InMemoryKeywordSearchService` | `ConcurrentHashMap` |
| `MemoryMetricsCollector` | `LongAdder` + `ConcurrentHashMap` |
| `RelevanceTracker` | `ConcurrentHashMap` |
| **pipelineExecutor（v2.0 新增）** | `SingleThreadExecutor`（守护线程 `auto-pipeline`），驱动自动总结与自动压缩 |
| `summarizationInProgress` | `ConcurrentHashMap<sessionId, Boolean>` 防重复触发 |

## 12. 关键设计决策

| 决策 | 理由 |
|------|------|
| 所有能力默认启用，LLM 功能根据基础设施自动检测 | 降低接入门槛，应用方只需提供 apiKey 即获得全部能力 |
| SPI default 方法扩展 | 不强制已有实现修改 |
| MemoryEntry 不可变 + with* 方法 | 线程安全，无副作用 |
| 衰退 = 降级而非删除 | 数据不丢失，可找回 |
| 容量检查只计 CORE 层 | ARCHIVED 不挤占 Prompt |
| SimpleEmbeddingService 用哈希 | 零外部依赖，开发即用 |
| 向量 metadata 存 content | 搜索结果可直接展示，无需二次查询 |
| OpenAIIntentSummarizer 手动解析 JSON | core 模块不强制依赖 Jackson |
| MemoryDecayEngine 用守护线程 | JVM 退出时自动回收，不阻塞 |
| ConflictDetectionMode=SYNC 仍为默认（可配置 ASYNC） | 保持兼容行为，ASYNC 可选 |
| RRF k=60 常数 | 业界标准，向量/关键词权重均衡 |
| RelevanceTracker BOOST_INTERVAL=5 | 避免频繁 I/O，批量提升 |
| MetricsCollector 用 LongAdder | 高并发低竞争，零 GC 压力 |
| getAllUserIds() default 返回空列表 | 向后兼容已有 MemoryStorage 实现 |
| 自动管道采用 ASYNC 模式（推荐默认） | 不阻塞对话主流程，总结和压缩在后台完成，用户体验更流畅 |
| 旧 boolean 开关静默忽略而非报错 | 平滑升级，已有 application.yml 无需修改即可运行 |
