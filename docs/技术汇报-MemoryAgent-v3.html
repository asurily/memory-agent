问题：开关爆炸

应用方的认知负担

v1.x 设计中，MemoryAgentConfig 包含 10 个 boolean 功能开关，全部默认关闭：

.securityScanEnabled(true)          // 安全扫描？死代码，从未检查
.autoMemoryExtractionEnabled(true)  // 需要手动开启
.semanticSearchEnabled(true)        // 需要手动开启
.decayEnabled(true)                 // 需要手动开启
.conflictDetectionEnabled(true)     // 需要手动开启
.hybridSearchEnabled(true)          // 需要手动开启
.autoImportanceLearningEnabled(true) // 需要手动开启
.metricsEnabled(true)               // 需要手动开启
.incrementalIntentEnabled(true)     // 需要手动开启
.autoPipelineMode(ASYNC)            // 需要手动设置

应用方根本不清楚哪些应该开启。这些都是本项目的能力，都应该具备，而不是交给应用方逐个控制。

解决方案：能力默认启用 + 自动检测

三条分类原则

分类原则能力
始终启用 纯内部功能，无外部依赖，零额外开销 安全扫描、语义搜索、混合检索、记忆衰退、重要性自学习、指标收集、增量意图、自动管道
有 LLM 时 依赖 LLM API 调用，无 LLM 则跳过 自动记忆提取、冲突检测
调优参数 保留阈值/维度等数值参数供精细控制 confidenceThreshold, embeddingDimension, decayHalfLifeDays, conflictDetectionMode 等

自动检测逻辑

// MemoryAgent 构造函数中
boolean hasLlm = builder.summarizer != null
    || (config.getLlmApiKey() != null && !config.getLlmApiKey().isEmpty());

// hasLlm == true  → 自动创建 MemoryExtractor + MemoryConflictDetector
// hasLlm == false → 这两个组件为 null，LLM 相关功能自动跳过

删除的 10 个开关

删除的开关原因新行为
securityScanEnabled 死代码（定义了但从未被检查） 始终启用
incrementalIntentEnabled 纯内部功能，有 graceful fallback 始终启用
metricsEnabled 纯内存 LongAdder，零开销 始终启用
autoPipelineMode (OFF/ASYNC/SYNC) 内部编排，不应暴露给应用方 始终 ASYNC
semanticSearchEnabled 有零依赖默认实现 (SimpleEmbeddingService) 始终启用
hybridSearchEnabled 有零依赖默认实现 (InMemoryKeywordSearchService) 始终启用
autoImportanceLearningEnabled 纯内存 RelevanceTracker 始终启用
decayEnabled 纯数学 CompositeDecayStrategy 始终启用
autoMemoryExtractionEnabled 依赖 LLM，但可通过 hasLlm 自动检测 有 LLM 时自动启用
conflictDetectionEnabled 依赖 LLM，但可通过 hasLlm 自动检测 有 LLM 时自动启用

新增：自动管道编排

设计目标

让 processUserMessage() + recordAssistantMessage() 成为真正的两步式自动管道，内部自动编排总结、记忆提取、压缩。应用方只需调两个方法，所有高级功能自动运转。

数据流（ASYNC 模式）

用户发消息
  → processUserMessage()
      → 记录消息、加载快照/画像/总结、推导增量意图
      → shouldSummarize() == true → 向 pipelineExecutor 提交异步总结任务
      → 返回 TurnContext（含 pendingSummarization Future）
  → 应用调用 LLM
  → recordAssistantMessage()
      → 记录助手消息
      → shouldCompress() == true → 向 pipelineExecutor 提交异步压缩任务
      → 返回
  [后台] 总结完成 → 自动记忆提取（有 LLM 时）→ 结果存入 Storage
  [后台] 压缩完成（如需要）
  下一轮 processUserMessage() 从 Storage 读取最新总结

关键实现

组件说明
pipelineExecutordaemon 单线程执行器（名称 auto-pipeline），负责所有异步任务
summarizationInProgressConcurrentHashMap.newKeySet()，防止同一 session 重复触发总结
pendingSummarizationCompletableFuture<IntentSummary> 挂在 TurnContext 上，调用方可选等待
autoCompressionContextWindowTokens压缩阈值，默认 8000 tokens，可通过 Builder 调优

架构设计演进

v1.0 — 基础能力

会话管理、意图总结、记忆 CRUD、安全扫描、上下文压缩。纯 SPI 架构，零框架依赖。

v1.1 — 高级功能（默认关闭）

自动记忆提取、语义搜索、记忆衰退、冲突检测、分层记忆。全部 boolean 开关控制。

v1.2 — 增强能力（默认关闭）

异步冲突检测、混合检索 RRF、重要性自学习、性能指标。继续增加 boolean 开关。

v2.0 — 能力默认启用 + 自动管道

本次重构。移除全部 10 个功能开关，能力根据基础设施自动启用。新增自动管道（ASYNC 后台总结+压缩），应用方只需 processUserMessage + recordAssistantMessage 两个方法。

关键设计决策

决策理由
所有能力默认启用，而非默认关闭 降低应用方认知负担，SDK 的能力应该开箱即用
LLM 功能通过 hasLlm 自动检测 不提供 apiKey 则 NoOpIntentSummarizer，LLM 功能自然跳过；提供 apiKey 则自动启用
自动管道始终 ASYNC 模式 ASYNC 不阻塞调用方主路径，是绝大多数场景的最佳选择。去掉 SYNC 模式简化代码
保留 conflictDetectionMode (SYNC/ASYNC) SYNC/ASYNC 是行为选择（是否阻塞写入），不是启用/关闭
旧 boolean 开关静默忽略 application.yml 中的旧配置不报错，平滑升级
SimpleEmbeddingService + InMemoryVectorStore 作为默认 零外部依赖，开发即用。生产环境通过 SPI 替换为真实实现
