解决的痛点

❌ 没有 Memory Agent 时

  • ❌ LLM 每次对话都是"失忆"状态，不知道用户之前说了什么
  • ❌ 多轮对话后 Token 爆炸，Context Window 溢出
  • ❌ 用户改了偏好（如从 MySQL 改 PostgreSQL），Agent 还在说旧信息
  • ❌ 长期积累后记忆只增不减，低价值信息挤占 Prompt 空间
  • ❌ 恶意用户注入提示词攻击，Agent 被操控

✅ 接入 Memory Agent 后

  • ✅ 跨会话保持用户记忆，Agent 始终了解用户背景
  • ✅ 三阶段上下文压缩，自动管理 Token 窗口
  • ✅ LLM 驱动的冲突检测，自动合并/替换矛盾信息
  • ✅ 时间衰减 + 频率评分，低价值记忆自动归档
  • ✅ 三层安全扫描：不可见字符 + 提示注入 + 数据泄露

// 1. 创建 Agent（提供 apiKey，所有能力自动就绪）
MemoryAgent agent = MemoryAgent.builder()
    .config(MemoryAgentConfig.builder()
        .llmApiKey("sk-xxx")

    .build();

// 2. 用户发消息（自动：记录消息 + 加载记忆 + 推导意图 + 后台总结）

// 3. 拿到上下文前缀，拼到你的 LLM Prompt 前面
String promptPrefix = ctx.getMemoryContextPrefix();

分层记忆架构

所有记忆自动分为三层，CORE 层始终注入 Prompt，ARCHIVED 通过语义搜索按需召回，RAW 保留原始记录。

L1 : CORE — 核心记忆
用户画像 + 最近的 Agent 笔记。captureSnapshot() 只取这一层。始终注入 LLM Prompt。
↓ 衰退评分 < 阈值 ↓    ↑ promoteMemory() 手动提升 ↑
L2 : ARCHIVED — 归档记忆
衰退降级的记忆、手动归档的记忆。不占 Prompt 空间，通过语义搜索找回。
↓ 按需检索 ↓
L3 : RAW — 原始记录
原始对话消息。可选检索。

联动机制

联动说明
记忆衰退 → 分层低分记忆自动从 CORE 降级到 ARCHIVED，不删除数据
重要性自学习 → 衰退高频检索的记忆获得更高评分，0.8+ 的记忆受保护不被淘汰
语义搜索 → 归档ARCHIVED 层记忆可通过语义搜索找回，提升后回到 CORE 层
冲突检测 → 写入新记忆写入前自动检测与已有记忆的矛盾，合并或替换
自动提取 → 写入LLM 总结后自动提取值得记忆的信息，安全扫描 + 去重后写入

memory-agent-spring-boot-starter

关键设计决策

决策理由
所有能力默认启用，LLM 功能自动检测降低应用方认知负担，SDK 能力开箱即用
不可变值对象 + with* 方法天然线程安全，无副作用
